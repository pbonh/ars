## Definition

Facts are variables that Ansible discovers automatically from a managed node before executing a play. They describe the host’s operating system, network interfaces, hardware, and other system attributes.

## How It Works

- **Gathering.** By default, Ansible runs the `setup` module at the start of each play to collect facts. This can be disabled with `gather_facts: false` if not needed.
- **Caching.** Facts can be cached between playbook runs using a cache plugin (JSON file, Redis, Memcached, etc.), avoiding repeated discovery overhead.
- **Custom facts.** Users can place static JSON or INI files in `/etc/ansible/facts.d` on the target host, or use the `set_fact` task to create runtime facts.
- **Magic variables.** Ansible also injects special variables such as `inventory_hostname`, `groups`, `hostvars`, and `play_hosts` that behave like facts but are generated by the engine.

## Key Parameters

- `ansible_facts` — the dictionary containing all discovered facts.
- `gather_facts` — play-level toggle.
- `fact_path` — path on the remote host for custom fact files.
- `cacheable` — attribute on `set_fact` that stores the fact in the fact cache.

## When To Use

Use facts when your playbook logic must adapt to the target system: installing the correct package manager, binding to the primary IP, skipping tasks on Windows, or branching based on OS family. Use custom facts to expose application-level metadata (e.g., which microservice role a host serves).

## Risks & Pitfalls

- **Fact gathering latency.** On large inventories, `setup` can add noticeable startup time; use fact caching or disable gathering if facts are not required.
- **Fact immutability.** Standard facts are read-only; `set_fact` creates new host-scoped variables rather than mutating `ansible_facts` directly.
- **Cache staleness.** Cached facts may become outdated if the system configuration changes; implement cache invalidation or periodic refresh.
